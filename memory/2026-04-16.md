### Architecture Patterns (what works across all apps)

**Standard app structure (all apps):**
- `app.py` monolith (Flask) — single file, all routes
- SQLite at `/data/` (Railway volume for persistence)
- `gunicorn app:app --bind 0.0.0.0:$PORT` start command
- `railway.json` for deploy config
- `/health` endpoint returning `{"status":"ok","db":"ok"}`
- WAL mode: `PRAGMA journal_mode=WAL` + `synchronous=NORMAL`
- Rate limiting: SQLite-based, 10 login attempts/min
- Session cookies: `httponly=True, samesite=Lax`

**Auth system (consistent across all apps):**
- SHA256 or bcrypt password hashing
- `login_required` decorator
- `admin_required` / `overseer_required` for admin routes
- `ADMIN_USER='admin'`, `ADMIN_PASS=os.environ.get('ADMIN_PASSWORD','admin123')`  ← NOTE: default is admin123 not admin1
- `session.permanent=True` + 8h lifetime
- `plan_required()` decorator for feature gating

**Multi-tenant pattern (Liberty Inventory + Consignment + others being upgraded):**
- Tenants = client stores under `/data/customers/<slug>/`
- `active_store_slug()` → checks `session['impersonating_slug']` or `session['store_slug']`
- `get_store_paths(slug)` → returns dict of paths for that tenant
- Overseer can impersonate any tenant

### Critical Lessons Learned (from research/ + sessions)

1. **Default admin password is `admin123` NOT `admin1`** — confirmed from app code. `admin1` only if `ADMIN_PASSWORD=admin1` set in Railway env vars
2. **Secret key persistence** — list-it-everywhere solved this correctly: persist SECRET_KEY to `/data/secret_key` file so sessions survive restarts. Other apps use env var fallback. If env var not set + redeploy → sessions invalidated
3. **`session=; Max-Age=0`** on login response = sessions being cleared = either wrong password OR secret key changed
4. **url_for() errors 500** — always double-check function name matches `url_for()` call. `product_view` vs `view_product` caused a 500
5. **Jinja `{% raw %}{% endraw %}`** — needed around CSS with `{}` braces inside `<style>` blocks in Flask templates. Removing causes 500.
6. **Railway volume** — all data at `/data/`. `start.sh` must `mkdir -p $DATA_DIR/uploads` etc before gunicorn starts
7. **CACHEBUST in Dockerfile** — bump `ARG CACHEBUST=<timestamp>` to force Railway to rebuild without cache
8. **OpenRouter vision models** — only use: `google/gemini-flash-1.5`, `google/gemini-2.0-flash-001`, `openai/gpt-4o`, `openai/gpt-4o-mini`, `anthropic/claude-3-haiku`, `anthropic/claude-3.5-sonnet`
9. **CSRF tokens** — the `_validate_csrf()` function exists in apps but is NOT automatically called. Must manually add `<input type="hidden" name="csrf_token" value="{{ csrf_token() }}">` to every POST form
10. **jay-portfolio branch is `master` not `main`** — always push to master for that repo

## 🌐 WEB DESIGN BRAIN UPDATE — Global Repo Scan (2026-04-16 ~01:35 UTC)

### The Web Design Landscape Right Now

**The dominant stack for modern SaaS UIs (2025-2026):**
- Tailwind CSS (94k★) — utility-first, THE standard
- shadcn/ui (112k★) — copy-paste component library, built on Radix UI primitives
- Radix UI (18k★) — unstyled accessible primitives (used under shadcn)
- DaisyUI (40k★) — Tailwind component library with themes
- Framer Motion — animation standard for React
- Floating UI (32k★) — tooltips, popovers, dropdowns

**For Flask/Jinja2 (no React, what WE use):**
- No npm, no build step — pure CSS + vanilla JS
- Can copy HyperUI components (hyperui.dev) directly — Tailwind v4 but paste-able
- animate.css (82k★) — just add CDN link, add class names
- lax.js (10k★) — beautiful scroll animations, 4kb, vanilla JS only
- AOS (Animate On Scroll) — fade/slide elements as user scrolls
- SweetAlert2 — beautiful popups (replaces ugly `confirm()` dialogs)

### What Top SaaS Landing Pages Have (Standard Structure 2026)

Every great SaaS landing page has ALL of these:
1. **Hero** — headline, subline, CTA button, social proof number ("1,200+ stores")
2. **Activity ticker / social proof bar** — "Mike in Dallas just won a $14,200 bid" scrolling
3. **Feature grid** — 3-4 features with icons, concise descriptions
4. **How it works** — numbered steps (3 steps max)
5. **Testimonials** — 3 cards, photo + name + quote + star rating
6. **Pricing** — 2-3 tiers, middle tier highlighted, all include 14-day trial
7. **FAQ accordion** — 5-7 questions, collapsible
8. **Final CTA** — "Start Free Trial" with urgency

**What makes them feel premium (micro-details):**
- Gradient text on headlines (`background-clip: text`)
- Glassmorphism cards (`backdrop-filter: blur(10px)`)
- Subtle grain texture on hero bg
- CSS custom properties for all colors (easy theming)
- `transition: all 0.2s` on interactive elements
- Box shadow with brand color (`0 4px 14px rgba(99,102,241,0.3)`)
- Animated gradient border on CTA button
- Number counter animations (0 → 1,200 on scroll)
- Staggered fade-in on feature cards

### New Design Patterns Identified

**Neobrutalism (trending per HyperUI):**
- Bold black borders (2-3px solid black)
- Flat bright colors (yellow, pink, lime)
- Offset box shadows (`4px 4px 0 #000`)
- Good for: bold CTAs, pricing cards, alert banners
- NOT for: professional SaaS dashboards

**Glassmorphism (mainstream now):**
```css
.glass-card {
  background: rgba(255, 255, 255, 0.08);
  backdrop-filter: blur(12px);
  -webkit-backdrop-filter: blur(12px);
  border: 1px solid rgba(255, 255, 255, 0.15);
  border-radius: 16px;
}
```
Use on: dark hero sections, modals, stat cards

**Gradient mesh backgrounds (very 2026):**
```css
.hero-bg {
  background:
    radial-gradient(ellipse 80% 50% at 50% -20%, rgba(99,102,241,0.3), transparent),
    radial-gradient(ellipse 60% 40% at 80% 50%, rgba(168,85,247,0.2), transparent),
    #0f172a;
}
```

**CSS-only scroll snap sections:**
```css
.snap-container { scroll-snap-type: y mandatory; overflow-y: scroll; height: 100vh; }
.snap-section   { scroll-snap-align: start; height: 100vh; }
```
