## Bugs fixed tonight (FloodClaim Pro)

### 1. AI Estimate button — hardcoded stale willie token
- Frontend had `Authorization: Bearer S7LroZDvJSqz...` hardcoded in fetch calls
- Token no longer matched DB, causing 401 → HTML redirect → JS parse error
- Fix: removed Authorization header entirely (session cookie handles auth)

### 2. SECRET_KEY regenerating on every redeploy
- Without SECRET_KEY env var, app fell back to random token_hex → sessions invalidated
- Jay added SECRET_KEY to Railway env vars
- Also added stable hash fallback using RAILWAY_SERVICE_ID as last resort
- login_required now flashes "session expired" message instead of silent redirect

### 3. 500 on all claims with rooms (Jinja2 key collision)
- room_data dict used key 'items' which shadows Python dict's .items() method
- Jinja2 resolved rd.items to the method, not the value → TypeError
- Fix: renamed to 'line_items' and 'room_photos' throughout app.py + all templates

### 4. 500 on AI Estimate (sqlite3.Row .get() crash)
- Python 3.13 on Railway: sqlite3.Row doesn't support .get()
- Affected: photo.get(), claim.get() in ai_estimate() and willie_generate_estimate()
- Fix: added dict() conversion after all .fetchone()/.fetchall() where .get() is called

### 5. DB migration gap — photos table missing columns
- photos table created before room_id and ai_description columns were added
- No migration existed to add them to existing prod DB
- Fix: added migrate_photos_columns() function that runs on boot

### 6. Add Room not working
- Root cause was the 500 from Jinja2 key collision above
- Once that was fixed, Add Room worked correctly

### 7. Delete Room feature (new)

## Lessons learned
- Always test against real data with rooms/photos before pushing
- sqlite3.Row ≠ dict — must convert with dict() before using .get()
- Jinja2 dict key names must not match Python dict method names (items, values, keys, etc.)
- Run full test suite BEFORE pushing, not after
- Railway deploys take ~90s — test against live after confirming deploy
